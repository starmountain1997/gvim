# Commit-As-Prompt 参考文档

## 文件挑选原则

- 仅包含实现本需求所必需的代码、配置、测试、文档
- 排除格式化、依赖升级、生成文件等噪声变更
- 纯重命名或大规模格式化应作为独立提交
- 暂存中如含多个主题，请拆分为多次提交

## Commit 类型与标题前缀

### Context Prompt 提交

- 标题以 `prompt:` 开头
- 适用于需被转换为上下文 Prompt 的提交
- 示例：`prompt(auth): 支持 OAuth2 登录`

### 常规提交

- 沿用 Conventional Commits 前缀
- 类型：`feat`, `fix`, `docs`, `refactor`, `test`, `chore` 等
- 示例：`feat(auth): 添加 OAuth2 登录支持`

## 提交信息通用原则

### 有意义的命名

- 标题应简洁、明确，描述变更内容和目的
- 避免「修复 bug」「更新代码」等模糊词

### 结构化与规范化

- 采用 Conventional Commits 格式
- 包含作用域与简短主题
- 正文补充细节，便于自动生成变更日志

### 解释 Why 而非列举 What

- 重点说明动机或背景
- 而不仅仅是修改了哪些文件

## 高质量提交最佳实践

1. **结构化与聚合**：一次提交聚焦单一主题；大型变更可拆分多步
1. **深入 WHY**：关联业务目标、用户需求或缺陷编号
1. **具体 HOW**：描述整体改动策略、验证方式、风险提示
1. **清晰语言**：使用英文祈使句，避免模糊词
1. **自动化与追溯**：正文引用 Issue/PR 编号
1. **上下文完整性**：补充依赖或前置信息

## 动态上下文

使用 `!command` 语法注入动态信息：

```yaml
---
name: commit-summary
description: 生成提交摘要
---

当前分支提交历史：
!`git log --oneline -10`

待提交变更：
!`git diff --stat`
```
