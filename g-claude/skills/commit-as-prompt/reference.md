# Commit-As-Prompt 参考文档

## 为什么要用 WHAT/WHY/HOW 格式

一条好的提交信息应该能在没有代码的情况下独立传递价值。未来的 AI 会话或几个月后的你，读到这条提交时，应该立刻明白：发生了什么、为什么要做、用什么方式实现的。

- **WHAT** 描述变更本身——简洁、祈使式，不重复文件名
- **WHY** 说明动机——业务目标、用户需求、缺陷背景或合规要求；这是最容易被忽略、也最有价值的一行
- **HOW** 描述技术路径——整体策略、兼容性处理、验证方式；如有风险请在此标注

## 文件挑选原则

- 仅包含实现本需求所必需的代码、配置、测试、文档
- 排除格式化、依赖升级、生成文件等噪声变更
- 纯重命名或大规模格式化应作为独立提交
- 暂存中如含多个主题，请拆分为多次提交

## Commit 类型

### `prompt:` 提交

用于需要被 AI 工具作为上下文消费的提交：技能文件、提示词、供未来会话使用的参考文档。

格式：`prompt(<scope>): <主题>`

### 常规提交

沿用 Conventional Commits 前缀：`feat`, `fix`, `docs`, `refactor`, `test`, `chore`

## 高质量提交的检查清单

1. **单一主题**：一次提交只做一件事；大型变更拆分多步
2. **WHY 不能缺席**：不要只写「修复了 XX」，要写「因为 XX 导致 YY，所以…」
3. **具体而不啰嗦**：HOW 说策略，不列文件清单
4. **引用编号**：如有关联的 Issue / PR，在 HOW 或正文末尾注明
5. **语言统一**：标题与正文保持同一语言

## 不该出现在提交里的内容

- `console.log` / `print` / `debugger` 等调试语句
- 注释掉的旧代码
- 与本次需求无关的格式调整
- 临时文件、构建产物、`.DS_Store` 等
